# Consolidated Project Plan — devbymax Portfolio Landing Page

## 1. Overview

1.1. Build a freelancer portfolio landing page that sells web development to small businesses (cafes, salons, shops).

1.2. Positioning and goals
   1.2.1. Role: senior frontend developer plus conversion designer.
   1.2.2. Primary goal: sell, not only look nice.
   1.2.3. Audience: small business owners (cafes, salons, shops).
   1.2.4. Call to action: https://t.me/devbymax
   1.2.5. Page language: Russian.

1.3. Visual direction
   1.3.1. Dark premium theme.

1.4. Content rules
   1.4.1. No Lorem ipsum; use exact Russian copy from screenshots (or from content spec when available).
   1.4.2. No placeholder images; use CSS gradients instead.

1.5. Scope constraints
   1.5.1. Do not add new sections without asking.
   1.5.2. Do not change colors without explicit request.

## 2. Tech Stack and File Structure

2.1. Stack
   2.1.1. HTML, CSS, and vanilla JavaScript only.
   2.1.2. No React, Vue, or npm.
   2.1.3. Animations: GSAP from CDN or pure CSS.
   2.1.4. Fonts: Google Fonts CDN.

2.2. Repository layout
   2.2.1. `index.html`
   2.2.2. `css/style.css`
   2.2.3. `js/main.js`
   2.2.4. `assets/`
   2.2.5. `refs/`

2.3. Reference documentation (when present in repo)
   2.3.1. Full UI spec: `docs/ui_ux.md` (referenced from `project_plan.md`; root `ui_ux.md` exists as equivalent in this workspace).
   2.3.2. All copy: `docs/content.md` (referenced from `project_plan.md`; not present in this workspace at consolidation time).

## 7. Animation Timings

7.1. Micro interactions (hover): 0.2s ease.

7.2. Card hover: 0.3s ease.

7.3. Scroll reveal: 0.6s ease-out.

7.4. Counter: 1.5s ease-out.

7.5. Accordion: 0.35s ease.

7.6. Marquee: 40s linear infinite.

7.7. Avoid heavy animations (explicit project constraint).

## 8. Mobile, Responsive, and Layout Engineering

8.1. Mobile-first CSS; must work at 375px width.

8.2. Breakpoint example: hero switches at max-width 768px to single column.

8.3. Touch targets at least approximately 44 by 44 pixels; adequate spacing between targets.

8.4. Use relative units (percent, em, rem) where appropriate; CSS Grid and Flexbox for layout.

8.5. Breakpoints driven by content needs; test range of devices and orientations.

8.6. Thumb zones and touch-friendly controls for primary actions on small screens.

8.7. Off-canvas or overlay patterns for secondary navigation on small screens (full-screen nav overlay specified for header).

8.8. Progressive disclosure where it reduces clutter on small viewports.

8.9. Fluid layouts and media queries per best practice guidance in general UX doc.

## 9. Performance and Technical Requirements

9.1. Target PageSpeed score greater than 90.

9.2. Images in WebP format with lazy loading (when images are used).

9.3. Prefer CSS animations over JavaScript when possible for performance.

9.4. Consider critical CSS for above-the-fold content.

9.5. Optimize assets for mobile networks.

9.6. Lazy load non-critical resources.

9.7. Monitor and optimize Core Web Vitals: LCP, FID, CLS.

9.8. Responsive images with srcset and sizes when using raster images.

9.9. Lazy load images and video; responsive embedded media (e.g. iframes) via CSS.

9.10. Code splitting is listed in general UX doc for SPAs; for this static stack, keep payloads small and defer non-critical scripts if applicable.

## 10. Accessibility, Semantics, and Quality

10.1. Semantic HTML for structure and screen reader compatibility.

10.2. Alternative text for images and non-text content when images are present.

10.3. Full keyboard navigability for interactive elements.

10.4. Test with assistive technologies when possible.

10.5. Follow WCAG guidelines.

10.6. Intuitive navigation patterns; familiar components; clear CTAs.

10.7. Judicious use of motion; respect user preferences where applicable for reduced motion if implemented.

## 11. Information Architecture and Testing Practices

11.1. Organize content logically; clear labels and categorization in navigation.

11.2. Effective search functionality and sitemaps are general IA recommendations; apply if the product grows beyond a single landing page.

11.3. Browser devtools for responsive checks; test on real devices; usability testing across device types.

11.4. A/B testing, heatmaps, session recordings, and continuous iteration when measuring conversion after launch.

11.5. Documentation: maintain style guide and pattern documentation; user flows for complex interactions; organized design assets for team use.

## 12. Delivery and Engineering Practices (from .cursorrules)

12.1. Output complete files; do not use “rest unchanged” placeholders.

12.2. Output code directly when implementing; avoid unnecessary preamble in generated code responses.
